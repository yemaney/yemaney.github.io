Classes

Classes are defined by the class keyword. The basic pattern of a class in python looks like.


class MyClass:  # class keyword followed by name of class to be created

	def __init__(self, name: str): # multiple functions inside the body of the class
	    self.name = name

	def method(self):
		return self.name

# instantiating the class
my_name = 'Yemane'
my_class_instance = MyClass(my_name)

# using the class
name = my_class_instance.method()
print(name)
# Yemane

As shown above, classes in python are syntactically just a collection of related functions, underneath a class name. They serve as blueprints for the behavior and structure of objects. The actual object instances are created using the Class(arguments).

Python Data Model

everything is an object

The python data model is critical to understanding how objects work in python. Consider the code below.


arr = [1,2,3,4]
print(type(arr))
# <class 'list'>

We create a list and then ask python to print out its type. Predictably it tells us are list is of list type. But it also prints out the word class. Which is telling us arr is an object defined by a list class. Similar results occur when this is tried on on other basic types like integers and dictionaries. It this case objects such as lists are so common that the class boilerplate is automatically handled by the python interpreter.

special methods

Special methods in are a set of reserved names for class methods that have a naming convention that follows the pattern of __<name>__. The first special method we've seen is the __init__ method, which is implicitly called when the user instantiates a class by invoking the class name followed by round brackets like UserClass().

Example 1: Size of a container


class MySeq1:
	def __init__(self, *args):
		self.myseq = list(*args)

	def size(self):
		return len(self.myseq)

class MySeq2:
	def __init__(self, *args):
		self.myseq = list(*args)
	
	def __len__(self):
		return len(self.myseq)

arr = [1,2,3,4]
ms1 = MySeq1(arr)
ms2 = MySeq2(arr)

# Get number of elements in our object
print(ms1.size())
print(len(ms1))
# 4
# 4

If you were to create a custom sequence-like class, one function to implement is a method to return the number of elements in the object. This can be by either defining a custom method size or by implementing the __len__ special method.

Example 2: Operator Overloading

Continuing with the example of implementing a custom sequence-like class, you may want to implement custom math operations for the class. In this case we will implement the addition operator + such that adding an integer to the class instance will add the integer to every element. MySeq([1,2,3]) + 1 = MySeq([2,3,4]) 


class MySeq1:
	def __init__(self, *args):
		self.myseq = list(*args)

	def add(self, i):
		return MySeq1((elem+i for elem in self.myseq))

class MySeq2:
	def __init__(self, *args):
		self.myseq = list(*args)
	
	def __add__(self, i):
		return MySeq2((elem+i for elem in self.myseq))

arr = [1,2,3,4]
ms1 = MySeq1(arr)
ms2 = MySeq2(arr)

# Perform the operation
ms1 = ms1.add(1)
ms2 = ms2 + 1

# Print new values
print(ms1.myseq)
print(ms2.myseq)
# [2,3,4,5]
# [2,3,4,5]

Similar to the first example we can either create a custom method, or use a special method. Notice, the logic in the functions is identical, it is only the name that differs.

These special methods (__<name>__) are not meant to be explicitly called by a user, but rather implicitly by the python interpreter. They are understood by the interpreter to be called when the correct built-in function like or operator is used. This is why the __add__ method allowed the class to be compatible with the + operator..

Some patterns

The first pattern is that names of the special methods often tied to the operation they are responsible for. And these inner class methods often map to a top level function or operator that a human will use to communicate with the object. Like __len__ maps to the len() function.

The second pattern is that the implementation of the special methods often uses the functions or operators they delegate to.

It's generally best practice to use special methods instead of custom methods when possible, and to follow the pattern described. This will lead to will behaved pythonic classes that work the same way as builtin classes do, which will allow the custom classes to integrate easily with the standard library. It will also make the custom class more understandable and predictable to other people who use your objects, because they will act the same normal python objects do.
